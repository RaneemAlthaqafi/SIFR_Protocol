\begin{abstract}
Current AI-agent systems rely heavily on web-era protocols and ad hoc message formats, including HTTP APIs, JSON-RPC, Server-Sent Events, WebSockets, plain JSON function calls, and free-form natural language messages. These mechanisms are useful and widely deployed, but they were not designed as a native substrate for autonomous agents that must prove identity, exchange constrained capabilities, invoke tools safely, and preserve auditable reasoning and execution lineage. This paper presents SIFR, Secure Interchange for Federated Reasoning, a protocol design that binds signed typed frames, explicit capability grants, replay and revocation checks, sandboxed tool execution, and content-addressed audit lineage into one executable communication model.

The evaluated implementation includes encrypted-at-rest key management with rotation and revocation metadata (Argon2id KDF, AES-256-GCM with kid-bound AAD); DID resolution for did:web (over HTTP/HTTPS) and a local did:sifr method; VC-inspired capability credentials with mutation defenses; signed capability revocation with on-load signature re-verification; replay protection bound to (sender, session, message\_id) over a sliding timestamp window; sandboxed WASM tool execution via wasmtime with no WASI imports linked and per-call fuel limits; a QUIC transport via aioquic with ALPN-negotiated connections evaluated on localhost loopback; an 11-attack controlled adversary evaluation; and a TLA+ model of the capability authorization state machine, model-checked with TLC across 276{,}205 distinct states under 7 invariants. The end-to-end vertical slice runs over QUIC on localhost and exits with the expected output (\texttt{Result: 5}, \texttt{Demo completed successfully.}). Measurements show signed SIFR action frames of 411 bytes versus 53 bytes for a plain JSON action, Ed25519 signing at 0.0314 ms, verification at 0.0615 ms, sign+verify+DAG append at 0.1104 ms, DID:sifr cold resolution at 0.068--0.077 ms, QUIC loopback round-trip latency at 0.3831 ms, WASM warm execution at 40.6 us, and reject latency for the 11 attack classes between 1.0 us and 1124.2 us. We make explicit non-claims: no W3C VC compliance, no cryptographic proof, no HSM- or PKI-grade key isolation, no distributed revocation, no arbitrary-untrusted-code WASM safety, and no production-deployment readiness. These results support the central claim that structured, signed, capability-checked agent communication can be implemented with measurable overhead while preserving verifiable authorization and audit evidence.
\end{abstract}

\section{Introduction}
Large language model (LLM) agents are increasingly deployed as systems that reason, call tools, retrieve context, delegate subtasks, and coordinate with other agents. In practice, these interactions are often represented as text or JSON carried over web infrastructure. A typical system may encode a tool call as a JSON object, send it over HTTP, stream partial results through Server-Sent Events, or pass a natural language transcript from one agent to another. This approach is pragmatic, but it leaves important semantics outside the protocol layer: whether a message is a thought, action, observation, tool invocation, critique, error, or final result; whether the sender is authorized to request the action; whether the permission is bounded and revocable; and whether an auditor can later verify the lineage of the interaction.

Recent interoperability work has begun to address adjacent problems. The Model Context Protocol (MCP) standardizes how model hosts connect to tools, prompts, and resources over JSON-RPC \cite{mcp2024spec,anthropic2024mcp}. Agent2Agent (A2A) targets heterogeneous agent discovery and task exchange \cite{a2a2025spec,google2025a2a}. IBM/BeeAI's Agent Communication Protocol (ACP) explored HTTP-native agent messaging and has become closely related to the A2A ecosystem \cite{beeai2025acp,ibm2025acp}. The Agent Network Protocol (ANP) explores identity, encrypted communication, and protocol negotiation for open agent networks \cite{chang2025anp,anp2025didwba}. These efforts are valuable, but a research gap remains: a minimal, reproducible, agent-native frame design that binds typed semantics, cryptographic integrity, fine-grained capabilities, and content-addressed audit lineage in one executable protocol.

This paper investigates that gap through SIFR, Secure Interchange for Federated Reasoning. SIFR is not presented as a replacement for HTTP, MCP, A2A, ACP, or ANP. Instead, the paper asks a narrower research question: what does a verifiable agent communication protocol look like if every meaningful action is represented as a signed typed frame, authorized by a signed capability grant, checked for replay and revocation, executed through a bounded tool interface, and recorded in an audit DAG?

The contributions of this work are:
\begin{itemize}
\item A protocol model for signed, typed AI-agent frames with explicit action, observation, result, error, critique, capability, and TensorFrame message types.
\item A minimal capability layer that binds issuer, subject, allowed actions, resource scope, expiration, budget, and delegation constraints.
\item A content-addressed audit DAG for verifiable message lineage.
\item An executable Python implementation of the complete two-agent calculator workflow over QUIC and WASM.
\item Unit tests, negative authorization and tampering tests, benchmark scripts, raw benchmark outputs, generated figures, and IEEE-style paper source.
\end{itemize}

\section{Background and Related Work}
\subsection{Agent Communication Languages}
Structured agent communication predates LLMs. KQML represented agent messages using performatives for knowledge and query exchange \cite{finin1997kqml}. FIPA ACL specified message structures and communicative acts such as request and inform \cite{fipa2002aclmessage,fipa2002communicativeacts}. These systems established that agent messages benefit from explicit semantics. However, they were designed for symbolic multi-agent systems rather than modern LLM agents that call external tools, consume untrusted retrieved content, and execute generated workflows.

\subsection{Tool-Using LLM Agents}
ReAct showed that interleaving reasoning traces and actions can improve language model performance on tasks requiring environment interaction \cite{yao2023react}. Toolformer demonstrated that language models can learn when and how to call external tools \cite{schick2023toolformer}. TalkHier argues that structured communication and hierarchical refinement can improve multi-agent systems \cite{wang2025talkhier}. These works support the premise that agent communication should distinguish reasoning, actions, and observations. They do not, by themselves, define a cross-agent security or audit protocol.

\subsection{Modern Agent Interoperability Protocols}
MCP addresses the integration of models with external context and tools \cite{mcp2024spec,anthropic2024mcp}. A2A focuses on agent discovery, task exchange, modality negotiation, and enterprise interoperability \cite{a2a2025spec,google2025a2a}. ACP explored rich messages, streaming, long-running tasks, and shared state in an HTTP-native agent communication layer \cite{beeai2025acp,ibm2025acp}. ANP proposes identity-oriented agent networking with protocol negotiation \cite{chang2025anp}. SIFR differs in scope: it focuses on signed frame semantics, capability enforcement, and audit DAG lineage. Protocol adapters could transport SIFR frames through existing ecosystems, but deployed interoperability is outside the present evaluation.

\subsection{Security, Identity, and Provenance}
NIST Zero Trust guidance emphasizes per-request access decisions rather than implicit trust in the network perimeter \cite{rose2020zerotrust}. DID Core and Verifiable Credentials provide standards for decentralized identifiers and cryptographically verifiable claims \cite{w3c2022did,w3c2025vc}. SIFR uses DID-style identity resolution and VC-inspired signed capability credentials to bind an agent identity to constrained actions. Merkle DAGs provide content-addressed structures where node identifiers are derived from content and links \cite{ipfs2025merkledag}. Supply-chain provenance work such as SLSA motivates attestable workflows and reproducible evidence \cite{slsa2025spec}.

\subsection{Transport and Sandboxing}
QUIC provides multiplexed streams and integrated transport security \cite{iyengar2021quic}, and HTTP/3 carries HTTP semantics over QUIC \cite{bishop2022http3}. SIFR evaluates a QUIC backend for signed agent frames on localhost loopback. WebAssembly and WASI motivate portable sandboxing for tool execution \cite{w3c2019wasm,wasi2024interfaces}; SIFR uses wasmtime to execute a bounded calculator module without linking WASI imports.

\subsection{Agent-Specific Risks}
Prompt injection research shows that LLM-integrated applications can be manipulated through untrusted content, including content that indirectly affects tool behavior \cite{greshake2023indirectpi,liu2024promptinjection}. OWASP LLM and MCP security guidance identifies tool poisoning, command injection, prompt injection, and unsafe tool execution as major risks \cite{owasp2025llmtop10,owasp2026mcptop10}. Embedding inversion and KV-cache leakage work cautions against assuming that latent or vector representations are private \cite{morris2023embeddingprivacy,luo2026shadowcache}. These risks motivate SIFR's explicit authorization and audit design.

\section{Problem Statement}
The central problem is not that HTTP or JSON are inadequate as byte carriers. The problem is that common agent communication stacks treat agent messages as application-defined objects whose semantics and authorization are often enforced inconsistently outside the communication layer. This creates at least five gaps.

First, message intent is ambiguous. A model-generated string may contain reasoning, a command, a partial observation, a critique, or a final answer, but the transport does not enforce a typed distinction. Second, identity and authorization are often ambient. API keys or bearer tokens may grant broad authority without binding the individual agent action to a constrained capability. Third, audit logs are frequently append-only text traces rather than cryptographically verifiable lineage structures. Fourth, tool execution may occur without a protocol-level proof that the action was allowed. Fifth, compact non-text channels such as embeddings or latent representations require careful framing and privacy warnings.

SIFR therefore tests whether a minimal protocol can bind the following invariant: every meaningful action is a signed typed message, checked against a signed capability, executed through a bounded tool interface, and recorded in a content-addressed audit DAG.

\section{Design Goals and Hypotheses}
The evaluation considers six hypotheses:

\textbf{H1:} Structured signed agent messages provide clearer audit semantics than generic JSON messages.

\textbf{H2:} Capability grants can prevent unauthorized actions in a minimal two-agent workflow.

\textbf{H3:} An audit DAG can detect missing parents and message tampering.

\textbf{H4:} Signed SIFR frames introduce measurable overhead, but local Ed25519 cost is small enough for integrity-sensitive agent workflows.

\textbf{H5:} TensorFrame base64 binary encoding can be more compact than naive JSON-list encoding for numeric vectors.

\textbf{H6:} A transport abstraction allows local testing while preserving an implementation boundary for QUIC transport.

\subsection{Message Envelope}
Every SIFR message contains a protocol version, message identifier, session identifier, type, sender identifier, receiver identifier, UTC timestamp, parent CIDs, optional capability identifier, payload, and optional signature. The signature is computed over the canonical message without the signature field. The implemented message types are Hello, CapabilityOffer, CapabilityGrant, Thought, Action, ToolUse, Observation, Result, Critique, Error, and TensorFrame.

\subsection{Canonicalization and Signatures}
The implementation canonicalizes messages by removing the signature field, sorting JSON keys, using compact JSON separators, and encoding UTF-8. Ed25519 signatures are generated and verified using the \texttt{cryptography} library. Verification fails if the payload, sender, or any signed field changes after signing.

\subsection{Capabilities}
A CapabilityGrant payload binds an issuer, subject, allowed actions, resource scope, issue time, expiration time, call budget, payload-size budget, and delegation constraint. The action verifier checks the grant signature, issuer consistency, subject match, action membership, expiration, payload size, call budget, and delegation policy before tool execution.

\subsection{Audit DAG}
Each accepted message can be added to an audit DAG. The CID is computed as SHA-256 over canonical message bytes without the signature field. Nodes record CID, message id, message type, sender, receiver, parents, timestamp, and signature-valid status. The DAG verifier detects missing parents, invalid signature flags, and stored-message mutation.

\section{Security Model and Threat Analysis}
SIFR assumes that each verifier can resolve the correct public key for the relevant agent through the configured key resolver. The evaluated security goal is to reject tampered frames, unsigned frames, wrong-key verification, expired grants, wrong-subject grants, unauthorized actions, over-budget actions, replayed messages, revoked grants, malformed frames, and audit-DAG mutation before unauthorized tool execution occurs.

Listing~\ref{lst:verify} summarizes the implemented verifier logic. The key point is that signature verification and capability authorization are separate checks: a correctly signed Action can still be rejected if it lacks the right grant.

\begin{lstlisting}[caption={Capability and DAG verification sketch for SIFR.},label={lst:verify}]
authorize(action, grant, issuer_pubkey, store):
  verify_signature(action, sender_pubkey)
  verify_signature(grant, issuer_pubkey)
  require grant.payload.issuer == grant.sender_id
  require action.capability_id == grant.capability_id
  require grant.subject == action.sender_id
  require action.payload.action in grant.actions
  require now_utc < grant.expires_at
  require size(action.payload) <= grant.max_payload_bytes
  require store.calls(grant.id) < grant.max_calls
  require not action.delegate_to or grant.allow_delegation
  store.consume(grant.id)
  return ALLOW

verify_dag(dag):
  for node in dag.nodes:
    require sha256(canonical(node.message)) == node.cid
    require node.signature_valid
    for parent in node.parents:
      require parent in dag.nodes
  return OK
\end{lstlisting}

The evaluation distinguishes protocol evidence from broad security claims. Signatures establish message integrity under the resolved public key, but do not by themselves establish institutional trust. Capability grants authorize specific actions and are checked for replay and revocation, but distributed revocation propagation and multi-process budget reservation are outside the present implementation. The calculator tool is bounded because it performs only a structured addition operation in WASM; this does not establish safety for arbitrary untrusted code.

\section{Security Evidence}
\label{sec:hardening}
Section~\ref{sec:hardening} presents the evaluated security mechanisms. Each subsection cites the implementation file, the test file with positive and negative cases, and the adversarial assertion that prevents an in-name-only implementation.

\subsection{Encrypted Key Management}
SIFR stores private keys at rest in a passphrase-encrypted JSON file (\texttt{sifr/key\_management.py}). The file format binds an Argon2id-derived~\cite{biryukov2016argon2} AES-256-GCM key per keystore. The kid is bound as additional authenticated data on each entry, so swapping ciphertexts between two entries fails on decryption. Tests in \texttt{tests/test\_key\_management.py} include the assertions \emph{file-does-not-contain-raw-private-key} and \emph{kid-AAD-binding-prevents-swap}.

\subsection{DID Resolution}
SIFR implements two DID methods behind a common \texttt{DidResolver} ABC (\texttt{sifr/did/}). \texttt{did:web}~\cite{didweb2024spec} fetches DID documents over HTTPS per the W3C method specification, with percent-encoded port handling. \texttt{did:sifr} reads documents from a local directory and is intended for tests and air-gapped scenarios. Both are exercised by an in-process loopback HTTP fixture (\texttt{tests/fixtures/did\_web\_server.py}) that proves did:web tests use real HTTP traffic. Negative tests check kid-not-in-document, controller-mismatch, path-traversal, and id-mismatch reject paths.

\subsection{VC-Inspired Capability Credentials and Revocation}
\texttt{sifr/credentials.py} implements signed credentials structured like W3C Verifiable Credentials (\texttt{@context}, \texttt{type}, \texttt{issuer}, \texttt{issuanceDate}, \texttt{expirationDate}, \texttt{credentialSubject}, \texttt{proof}). The proof is an \texttt{Ed25519Signature2020} over canonical JSON without the proof field. Negative tests demonstrate that mutating \texttt{credentialSubject.id}, \texttt{expirationDate}, the embedded \texttt{capability.actions}, or \texttt{proof.proofValue} after signing all fail verification, and that the verification-method DID must match the issuer DID. The implementation is VC-inspired rather than a full JSON-LD/URDNA2015 Verifiable Credentials conformance implementation.

\texttt{sifr/revocation.py} implements a registry of signed \texttt{CapabilityRevocation} SIFR messages with optional JSONL persistence. Entries are re-verified on load, so tampering with a stored record fails reload. \texttt{sifr/capabilities.py:authorize\_action} consults the registry before authorization; a revoked grant raises \texttt{UnauthorizedAction("REVOKED\_CAPABILITY")} (attack 4 in Table~\ref{tab:adversary}).

\subsection{Replay Protection}
\texttt{sifr/replay.py} maintains a per-process cache keyed on \texttt{(sender\_id, session\_id, message\_id)} with a sliding timestamp window (default 5 minutes). A negative test demonstrates that re-signing the same message with a different signature value still fails replay because the cache key does not include the signature. Optional SQLite persistence retains entries across restarts. Window-edge cases (stale, future, exact boundary) are tested.

\subsection{WASM Tool Isolation}
\texttt{sifr/wasm\_runner.py} uses \texttt{wasmtime}~\cite{wasmtime}-based execution of \texttt{wasm/calculator/calculator.wat}. The runner does not link any WASI imports; modules importing \texttt{wasi\_snapshot\_preview1} fail to instantiate. Each call uses a fresh \texttt{Store} with a fixed fuel budget (default 1{,}000{,}000), bounding compute and trapping infinite loops. The committed module is WAT (text), not opaque \texttt{.wasm}, so the 13-line source is inspectable.

Adversarial assertions in \texttt{tests/test\_wasm\_runner.py}: (i) a parity test of 8 input pairs proves Python and WASM produce bit-identical results; (ii) every successful call advances \texttt{last\_invocation\_evidence["fuel\_consumed"]} (a Python fall-through would never advance it); (iii) the \texttt{fs\_attempt.wat} fixture, which tries to import \texttt{path\_open}, fails to instantiate; (iv) the \texttt{looping.wat} fixture traps on fuel.

\subsection{QUIC Transport}
\texttt{sifr/transport/quic.py} provides a single-stream length-prefixed canonical-JSON transport over \texttt{aioquic}~\cite{aioquic} 1.3 with ALPN \texttt{sifr/0.2}. Self-signed cert generation for tests and demos is implemented in \texttt{sifr/transport/\_certs.py}; private keys are not committed. The trap-acceptance test asserts \texttt{isinstance(transport.quic\_connection, aioquic.quic.connection.QuicConnection)} AND \texttt{negotiated\_alpn == "sifr/0.2"} AND that a real \texttt{original\_destination\_connection\_id} was negotiated --- ruling out a TCP look-alike. All QUIC measurements and tests in v0.2 use localhost loopback.

Loopback RTT for sign+verify+DAG over LocalTransport is 0.0868 ms; over QUIC the same workload takes 0.3831 ms (\texttt{benchmarks/results/v0.2/quic\_latency.csv}); the difference reflects QUIC handshake amortization, framing, and asyncio scheduling overhead.

\subsection{TLA+ Formal Model}

The TLA+ model is a bounded safety model rather than a cryptographic proof, liveness proof, or refinement proof. It captures the specification of authorization; the adversary tests in \texttt{tests/test\_network\_adversary.py} connect the modeled invariants to implementation-level rejection behavior.

\subsection{v0.3 formal model strengthening}
The v0.2 model checked seven invariants on a state machine with one issuer and one kid. The v0.3 model adds explicit \texttt{Issuers} and \texttt{Kids} to the state and two new invariants:
\begin{itemize}
\item \texttt{NoConsumeWithWrongIssuer} (I8): every consume references the issuer the cap was issued under.
\item \texttt{NoConsumeWithRevokedKey} (I9): every consume's kid is not in the \texttt{revokedKids} set.
\end{itemize}
The bounded model uses \texttt{Caps=\{c1\}}, \texttt{Subs=\{alice,bob\}}, \texttt{Issuers=\{alice,bob\}}, \texttt{Acts=\{add,multiply\}}, \texttt{Msgs=\{m1,m2\}}, \texttt{Kids=\{k1,k2\}}, \texttt{MaxCalls=2}. TLC explores 11{,}601 distinct states at depth 7 and reports no error for all 9 invariants. Two further invariants (I7 \emph{TamperedCredentialNeverAllowed} and I10 \emph{AuditTamperDetected}) are byte-level / DAG-level properties that the state machine deliberately abstracts away; they are validated only by implementation tests (\texttt{tests/test\_credentials.py}, \texttt{tests/test\_audit\_dag.py}, and adversary cases A05-A07 / A21).

\section{Discussion}
The implementation demonstrates that a compact vertical slice can bind communication semantics, signatures, capabilities, tool execution, and audit lineage. The central value of SIFR is that the protocol claim is executable and testable: the demo can be run, frames can be tampered with, grants can be expired, budgets can be exceeded, DAG parents can be removed, figures can be regenerated, and the resulting behavior can be inspected.

Concrete numbers support the implementation claim: the demo exchanges seven SIFR messages over a QUIC stream with sub-millisecond per-message overhead at the application layer; the WASM calculator runs in $\sim$40.6 $\mu$s warm and produces an audit fuel-consumption counter that distinguishes WASM execution from a Python fall-through; revocation lookups are below 0.2 $\mu$s in the measured cases; and the 11-attack adversary suite shows reject latencies bounded by 1124.2 $\mu$s even for the slowest path (replay, which subsumes a legitimate authorize+execute round). Figure~\ref{fig:capability_growth} summarizes the test and layer evidence.

\centering
\includegraphics[width=\linewidth]{figures/benchmark_v0_1_vs_v0_2.png}

\end{figure}

The results also show costs and trade-offs. Signed frames are larger than plain JSON actions. Ed25519 verification is slower than signing in the measured environment. DAG append adds additional local overhead. These costs may be acceptable for integrity-sensitive agent workflows, but the current evaluation is too small to justify broad performance claims. The TensorFrame experiment demonstrates encoding-size differences only; it does not demonstrate privacy, semantic quality, compression efficiency, or safe KV-cache sharing.

\section{Limitations}
The evaluated system has important scope boundaries. QUIC integration is exercised over localhost loopback with self-signed test/demo certificates; no real-network, NAT, packet-loss, multi-stream concurrency, or 0-RTT evaluation is presented. The credential layer is VC-inspired and does not implement JSON-LD context processing or URDNA2015 RDF normalization. The key store is not an HSM or enterprise PKI. Revocation propagation is local to the configured registry rather than distributed through gossip or consensus. WASM safety is evaluated for the calculator module and two adversarial fixtures, not arbitrary untrusted code or side-channel resistance. Fuel bounds compute steps, not wall-clock time. The TLA+ model checks bounded safety invariants, not cryptographic security, liveness, or full implementation refinement. The replay cache uses a per-process \texttt{dict}; its garbage collection is $O(n)$ per check and characterized as a known cost in \texttt{benchmarks/results/replay\_overhead.csv}. The benchmark and figure workflow is intentionally preserved for review, but v0.3 should replace it with a single fail-closed reproduction script and versioned manifests.

\section{Future Work}
Concrete next steps include distributed revocation gossip; multi-process concurrency-safe budget reservation; HSM/keyring-backed key storage as a default; richer formal modeling with cryptographic protocols via ProVerif or Tamarin; liveness analysis via TLC fairness annotations; wall-clock interruption for WASM via wasmtime epoch interruption; multi-tenant sandbox isolation; fuzz testing on message validation and authorization paths; protocol adapters bridging SIFR frames to MCP, A2A, ACP, and ANP carriers; and privacy-preserving TensorFrame designs informed by embedding inversion and KV-cache leakage research.

\section{Conclusion}
SIFR presents a narrow but complete vertical slice for structured, signed, capability-checked AI-agent communication: encrypted-at-rest key management; DID resolution; VC-inspired credentials with mutation defenses; signed revocation; replay protection; sandboxed WASM execution; QUIC transport; an 11-attack adversary suite; and a TLA+ model checked against 276{,}205 states under 7 invariants. The end-to-end demo exits 0 with the expected output. The system remains deliberately scoped, but it provides a concrete, testable baseline for agent-native protocol research.
