\begin{abstract}
Agentic AI systems are rapidly becoming service-oriented systems: agents discover tools, call remote capabilities, delegate subtasks, stream observations, and coordinate with other agents. Yet much of today's agent communication is still encoded as free-form text or ad hoc JSON over web-era interfaces. Those carriers are useful, but they do not provide agent-native semantics for typed intent, cryptographic sender binding, delegated authorization, replay resistance, revocation, safe tool invocation, and auditable reasoning lineage. This paper presents SIFR, Secure Interchange for Federated Reasoning, a protocol and artifact for signed, typed, capability-checked communication among AI agents.

SIFR represents agent communication as canonical signed frames, authorizes actions through constrained capability grants, rejects replayed and revoked requests, records content-addressed audit-DAG lineage, and invokes tools only after authorization. The artifact includes a Python implementation, QUIC transport, WASM calculator execution, DID-style key lookup, SIFR Capability Credentials, benchmark scripts, adversarial tests, a bounded TLA+ model, and a Tamarin symbolic model. The evaluated vertical slice runs a two-agent service workflow: hello, capability offer, capability grant, signed action, authorization, tool execution, signed observation, and audit verification. Empirical measurements show a signed action frame of 411 bytes versus a 53-byte plain JSON action, Ed25519 signing at 0.0314 ms, verification at 0.0615 ms, sign+verify+DAG append at 0.1104 ms, QUIC loopback round-trip at 0.42 ms, and Docker/NetEm QUIC behavior under 20 ms delay and 1--5\% loss. The security evidence includes 274 implementation tests, a 30-case adversarial rejection suite, standard cryptographic test vectors, process-shared replay and revocation tests, hardened WASM adversarial fixtures, runtime trace-conformance checks, TLC verification of 9 invariants over 11,601 states, and Tamarin verification of five symbolic lemmas: authentication, authorization, replay resistance, revocation safety, and tool-safety. SIFR is not a production deployment claim; rather, it is a proof-carrying research artifact showing how a native protocol for AI-agent services can bind structure, authorization, execution, and audit evidence into one reproducible system.

\keywords{AI agents \and Web services \and service protocols \and capability security \and formal verification \and Tamarin \and TLA+ \and audit logs}
\end{abstract}

\section{Introduction}

The next generation of Web services is becoming agentic. Instead of a human client manually invoking a fixed endpoint, an AI agent may discover a service, negotiate capabilities, call tools, inspect observations, delegate subtasks, and return a result to another agent. This shift is visible in current work on tool-using language models, agent interoperability, model-context protocols, and agent-to-agent systems. It also aligns with the theme of Agentic AI as a Service: agents are becoming service clients, service providers, and service orchestrators.

The communication layer has not fully caught up. Many deployed systems still treat an agent message as text, a chat transcript, or a loosely structured JSON object carried by HTTP, JSON-RPC, Server-Sent Events, or WebSockets. These mechanisms are valuable transport and integration tools, but they leave important agent semantics outside the protocol layer. A verifier often has to infer whether a message is a thought, an action, an observation, a critique, a result, or an error. Authorization may be encoded as an API key or bearer token rather than a narrow delegation tied to a specific agent, action, resource, time interval, and budget. Tool execution may occur before a durable audit trail can prove which signed request caused it.

This paper studies a concrete question: what would an agent-native service protocol look like if typed semantics, signatures, capability grants, replay protection, revocation, tool execution, and audit lineage were protocol-level properties rather than application conventions? We answer through SIFR, Secure Interchange for Federated Reasoning. SIFR is a protocol design and reproducible artifact for secure interchange among federated AI agents. The central design principle is simple: every meaningful action is a signed typed frame, authorized by a signed capability, checked for replay and revocation, executed through a bounded tool interface, and recorded in a content-addressed audit DAG.

SIFR does not claim to replace HTTP, MCP, A2A, ACP, ANP, or QUIC. Instead, it isolates a missing layer: a signed and auditable agent frame model that can be carried over a local transport, QUIC, or future interoperability adapters. This separation is useful for Web-service settings because it lets service providers reason about agent intent and authority independently of the byte carrier.

The contributions are:
\begin{itemize}
  \item A protocol architecture for typed, signed, capability-checked AI-agent service messages.
  \item A proof-carrying artifact that links protocol claims to code, tests, TLA+ invariants, and Tamarin lemmas.
  \item A complete implementation of a two-agent service workflow over QUIC with WASM-backed tool execution and audit-DAG lineage.
  \item A controlled adversarial evaluation showing that unauthorized, replayed, revoked, tampered, malformed, and audit-breaking actions are rejected before tool execution.
  \item Benchmarks quantifying payload size, signing and verification cost, local latency, QUIC behavior, capability enforcement, replay overhead, revocation lookup, DID resolution, credential verification, and WASM execution.
\end{itemize}

\section{Background and Related Work}

\subsection{Agent Communication}

Structured agent communication predates large language models. KQML and FIPA ACL introduced performatives and communicative acts for symbolic multi-agent systems \cite{finin1997kqml,fipa2002aclmessage,fipa2002communicativeacts}. Their central lesson remains relevant: agent messages benefit from explicit semantics. Modern LLM agents, however, introduce new service-level requirements. They call external tools, consume untrusted context, act under delegated authority, and produce traces that should be auditable after execution.

ReAct showed that interleaving reasoning and acting can improve language-model interaction with environments \cite{yao2023react}. Toolformer showed that models can learn to use tools through self-supervised signals \cite{schick2023toolformer}. Recent structured multi-agent work such as TalkHier argues that structured communication can improve coordination \cite{wang2025talkhier}. SIFR builds on the idea that actions and observations should be explicit, but focuses on protocol-level integrity and authorization.

\subsection{Agent Interoperability Protocols}

The Model Context Protocol (MCP) standardizes how model hosts connect to tools, resources, and prompts over JSON-RPC \cite{mcp2024spec,anthropic2024mcp}. Agent2Agent (A2A) targets heterogeneous agent discovery and task exchange \cite{a2a2025spec,google2025a2a}. IBM/BeeAI's Agent Communication Protocol explored HTTP-native messages, streaming, and long-running tasks \cite{beeai2025acp,ibm2025acp}. The Agent Network Protocol studies identity-oriented networking for open agent systems \cite{chang2025anp,anp2025didwba}. These efforts solve important interoperability problems. SIFR targets a narrower and complementary question: how to make each agent action a signed, typed, authorized, replay-checked, revocation-aware, and audit-linked protocol event.

\subsection{Security, Identity, and Provenance}

Zero Trust architecture emphasizes resource-specific, per-request decisions instead of implicit network trust \cite{rose2020zerotrust}. DID Core and Verifiable Credentials specify decentralized identifiers and verifiable claims \cite{w3c2022did,w3c2025vc}. SIFR uses DID-style key binding and SIFR Capability Credentials whose JSON shape resembles a Verifiable Credential, but it does not claim W3C VC compliance. Its credential layer signs constrained capability claims over canonical JSON rather than using JSON-LD processing or RDF canonicalization.

Content-addressed structures such as Merkle DAGs support tamper-evident lineage \cite{ipfs2025merkledag}. Supply-chain provenance work such as SLSA motivates reproducible evidence and verifiable artifact metadata \cite{slsa2025spec}. SIFR applies those ideas to agent messages: message CIDs are derived from canonical content and parent links form an audit DAG.

\subsection{Transport and Sandboxing}

QUIC provides multiplexed, secure transport over UDP \cite{iyengar2021quic}; HTTP/3 maps HTTP semantics onto QUIC \cite{bishop2022http3}. SIFR implements a QUIC backend using aioquic \cite{aioquic}, while retaining a transport abstraction so the message and authorization layers can be tested independently. WebAssembly and WASI motivate portable sandboxing \cite{w3c2019wasm,wasi2024interfaces}. The artifact uses wasmtime \cite{wasmtime} for a bounded calculator module with no WASI imports and fuel limits.

\subsection{Agent Threats}

Prompt injection research shows that untrusted content can manipulate LLM-integrated applications and tool behavior \cite{greshake2023indirectpi,liu2024promptinjection}. OWASP guidance identifies prompt injection, tool misuse, tool poisoning, and unsafe execution as risks for LLM and MCP-style systems \cite{owasp2025llmtop10,owasp2026mcptop10}. Embedding and KV-cache leakage work cautions that vector or latent representations can expose information \cite{morris2023embeddingprivacy,luo2026shadowcache}. SIFR does not solve all of these problems, but it gives service operators a protocol-level place to bind authority, execution, and audit evidence.

\section{Problem Statement}

Existing agent-service stacks commonly encode actions as application JSON or natural language over generic transports. This is practical, but it leaves five gaps.

\textbf{Typed intent gap.} A receiver may not know whether a message is a non-binding thought, a tool action, an observation, a critique, an error, or a final result.

\textbf{Delegated authorization gap.} A broad token may authorize too much. Agents need grants scoped by issuer, subject, action, resource, time, budget, and delegation policy.

\textbf{Replay and revocation gap.} A signed message can still be replayed if the receiver does not maintain message freshness state. A previously valid grant should fail once the verifier knows it has been revoked.

\textbf{Tool-safety gap.} Tool execution should be causally downstream of signature verification, capability authorization, replay checks, and revocation checks.

\textbf{Auditability gap.} Text logs are useful, but they usually do not provide content-addressed lineage that detects missing parents or message mutation.

SIFR's research claim is that these gaps can be addressed by a compact protocol layer whose claims are tied to executable tests, formal models, and benchmark evidence.

\subsection{Message Envelope}

A SIFR frame contains a version, message identifier, session identifier, type, sender identifier, receiver identifier, UTC timestamp, parent CIDs, optional capability identifier, payload, and optional signature. Implemented message types include \texttt{Hello}, \texttt{CapabilityOffer}, \texttt{CapabilityGrant}, \texttt{Thought}, \texttt{Action}, \texttt{ToolUse}, \texttt{Observation}, \texttt{Result}, \texttt{Critique}, \texttt{Error}, and \texttt{TensorFrame}.

The signature is computed over the canonical frame without the signature field. Canonicalization sorts keys, uses compact JSON separators, and encodes UTF-8. This gives deterministic signing and makes post-signature mutation detectable.

\subsection{Capability Grants}

A capability grant binds:
\[
  \langle issuer, subject, actions, resources, issued\_at, expires\_at, budget, constraints \rangle .
\]
The verifier checks issuer signature, issuer identity, subject identity, action membership, resource scope, expiration, payload-size budget, call budget, delegation constraints, replay status, and revocation status. A correctly signed action is not sufficient; the referenced grant must also authorize the action.

\subsection{Audit DAG}

Each accepted message can be inserted into an audit DAG. A node stores a CID, message id, type, sender, receiver, parents, timestamp, and signature-valid flag. The CID is SHA-256 over canonical message bytes without the signature field. The verifier rejects missing parents and detects mutation by recomputing CIDs. Figure~\ref{fig:dag} illustrates the vertical slice lineage.

\subsection{Workflow}

The evaluated workflow has two agents: a planner and an executor. The planner sends a signed \texttt{Hello}. The executor offers a calculator capability and issues a signed grant. The planner sends a signed \texttt{Action} requesting addition. The executor verifies the planner signature, validates the grant, checks replay and revocation state, appends \texttt{ToolUse}, executes the WASM calculator, returns a signed \texttt{Observation}, and verifies the audit DAG.

\section{Formal Security Analysis}

We use four categories. \emph{Bounded-proven} means TLC checked a finite TLA+ state space. \emph{Symbolic-proven} means Tamarin verified a lemma under a Dolev--Yao model. \emph{Tested} means implementation tests exercise positive and adversarial cases. \emph{Assumed} means a standard cryptographic or systems assumption.

\subsection{TLA+ Model}

The TLA+ model represents capabilities, subjects, issuers, actions, message ids, key ids, revocation, replay state, budgets, and accepted history \cite{lamport2002tla,tlatools}. TLC checks 9 invariants over 11,601 distinct states at depth 7:
\begin{itemize}
  \item type correctness;
  \item no unauthorized consumption;
  \item no wrong-subject consumption;
  \item no consumption after expiration;
  \item no consumption after revocation;
  \item no over-budget consumption;
  \item no replayed consumption;
  \item no wrong-issuer consumption;
  \item no consumption with a revoked key.
\end{itemize}
This is a bounded safety result, not a liveness proof and not an implementation-refinement proof.

\subsection{Tamarin Model}

The Tamarin model abstracts public-key signatures, grants, revocations, signed actions, verifier acceptance, and tool execution \cite{meier2013tamarin}. The Dolev--Yao adversary observes and replays wire messages but does not learn private keys of uncompromised principals. Tamarin Prover 1.7.1 verifies five lemmas:
\begin{itemize}
  \item \texttt{authentication}: an accepted action attributed to a subject was previously sent by that subject;
  \item \texttt{authorization\_required}: every accepted action has an issuing grant;
  \item \texttt{replay\_resistance}: the same accepted action is not accepted twice;
  \item \texttt{revocation\_safety}: accepted actions precede known revocation of the capability;
  \item \texttt{tool\_safety}: every tool run has a prior accepted action.
\end{itemize}

The replay lemma is intentionally documented as load-bearing. Without a replay-cache abstraction, a Dolev--Yao adversary can replay signed wire bytes. The model therefore includes an \texttt{accepted\_once\_per\_message} restriction representing the runtime \texttt{ReplayCache.check\_and\_record} invariant. The implementation tests exercise that cache directly.

\section{Discussion}

SIFR's main result is not that its prototype is faster than existing protocols. It is that agent-service communication can be made structurally verifiable without a large implementation. Typed frames clarify intent. Signatures bind content and sender identity. Capability grants narrow authority. Replay and revocation checks make old messages and withdrawn grants fail. Audit DAGs give a concrete lineage object. Tool execution becomes a downstream consequence of verified authorization.

For ICWS-style Web-service research, the most important implication is that agentic services need protocol-level authority and evidence. A future agent-service marketplace cannot rely only on broad API keys and natural language transcripts. Service providers need to know which agent requested which action under which grant, whether the grant was fresh and unrevoked, whether the same message had already been accepted, and which execution trace resulted.

The formal evidence also shapes the protocol design. The Tamarin replay result is a useful example: signatures alone do not prevent replay. The proof requires an accepted-once invariant corresponding to the implementation's replay cache. This makes the cache a first-class security component rather than an optimization.

\section{Cryptographic Assumptions and Test Vectors}
\label{sec:crypto-assumptions}

SIFR does not prove primitive security from scratch. We name each
assumption explicitly, cite the standardisation document, and validate
implementation integration against the standard's published test vectors.

We assume Ed25519~\cite{rfc8032} is existentially unforgeable under chosen
message attack in the random-oracle model; SHA-256~\cite{fips180-4} is
collision and second-preimage resistant; AES-256-GCM~\cite{nist80038d} is
an authenticated cipher under unique-nonce-per-key usage; and
Argon2id~\cite{rfc9106} resists offline password recovery at the
parameters specified in RFC 9106 §4. The repository file
\texttt{docs/crypto\_assumptions.md} states each assumption, the SIFR
construct that depends on it, and the consequence if the assumption is
broken.

The test file \texttt{tests/test\_crypto\_vectors.py} runs RFC 8032
§7.1 Ed25519 vectors (TEST 1, 2, 3), FIPS 180-4 SHA-256 short and
$10^{6}$-byte vectors, NIST SP 800-38D Test Cases 1 and 3 for AES-GCM,
and an RFC 9106 reference Argon2id call. Misuse-resistance tests confirm
that the wrong Ed25519 key fails verification, that a single-bit message
mutation invalidates the signature, that AES-GCM AAD/ciphertext/tag
tampering raise \texttt{InvalidTag}, and that Argon2id parameters are
recorded next to the tag and re-applied at decryption. SIFR's API does
not auto-generate AES-GCM nonces; nonce uniqueness is documented as a
caller responsibility, with a regression test that demonstrates the
keystream-XOR consequence of misuse.

\noindent
\textbf{Honest claim.} \emph{We do not prove primitive security; we
validate integration against standard vectors and rely on standard
cryptographic assumptions.}

\section{Limitations}

SIFR is a research artifact, not a production system. The following
limitations are explicit. Each item below is the v0.5 status; see
Table~\ref{tab:limitations-final} for the closed/narrowed/remaining
breakdown.

\textbf{Cryptographic assumptions.} As stated in
Section~\ref{sec:crypto-assumptions}: SIFR validates integration against
RFC 8032, FIPS 180-4, NIST SP 800-38D, and RFC 9106 vectors and relies on
the corresponding standard assumptions. SIFR does not prove primitive
security.

\textbf{Credential scope.} SIFR Capability Credentials are signed
Ed25519 grants whose JSON body resembles a W3C Verifiable Credential.
They are NOT W3C VC compliant: SIFR does not load JSON-LD contexts and
does not perform URDNA2015 RDF canonicalisation. SIFR ships
\texttt{SIFRStatusList2021}, a bitmap-based credential-status mechanism
modelled on but not interoperable with W3C StatusList2021.

\textbf{Identity scope.} SIFR supports \texttt{did:web}, \texttt{did:key},
and local \texttt{did:sifr} for Ed25519 keys encoded as
\texttt{publicKeyBase64}, \texttt{publicKeyMultibase}, or
\texttt{publicKeyJwk}. Other curves and other DID methods are out of
scope.

\textbf{Revocation and replay scope.} SIFR supports process-shared replay
through a SQLite-backed verifier state with WAL journaling and a
\texttt{PRIMARY KEY} uniqueness gate, and process-shared revocation
through a signed JSONL log re-verified at load time. SIFR does not
implement Byzantine consensus or global revocation propagation between
independent verifier deployments.

\textbf{WASM scope.} SIFR enforces a no-WASI, fuel-bounded, fresh-store,
memory-capped WASM policy tested against the calculator module and a
documented set of adversarial fixtures (filesystem, environment, socket,
infinite loop, memory-grow abuse, missing-export, unreachable trap).
SIFR does not claim arbitrary untrusted-code safety, side-channel
resistance, or multi-tenant isolation.

\textbf{Network scope.} The v0.5 harness adds a two-bridge Docker Compose
file, a seven-profile NetEm sweep (baseline, delay-20\,ms,
delay-100\,ms, loss-1\,\%, loss-5\,\%, jitter, bandwidth-10\,Mbit/s)
and a Python driver
(\texttt{benchmarks/bench\_quic\_network.py}). Evaluation remains
single-host emulated; SIFR does not claim multi-host, NAT-traversal,
mobile, or Internet-scale measurement.

\textbf{Formal scope.} TLA+ invariants are bounded-proven by TLC. SIFR
ships an operator-runnable Apalache configuration
(\texttt{formal/apalache.cfg}) for symbolic bounded re-checking, but
does not claim an Apalache proof unless the operator runs it and commits
the log. Tamarin abstracts cryptographic primitives and
includes a documented replay-cache restriction. SIFR adds runtime
\emph{trace-checked} conformance via
\texttt{sifr/trace\_conformance.py}: the same TLA+ invariants are
re-evaluated over traces emitted by the Python implementation, and
nine counterexample traces are rejected by the checker to prove
sensitivity. SIFR does not claim a Coq/TLAPS refinement proof from
Python to the formal models.

\section{Conclusion}

SIFR argues that agentic AI services need a native communication layer that treats messages as signed, typed, authorized, replay-checked, revocation-aware, and auditable protocol events. The artifact demonstrates a complete vertical slice: a two-agent service workflow over QUIC, capability-checked WASM tool execution, signed observations, and audit-DAG verification. The evidence is deliberately multi-layered: implementation tests, adversarial tests, benchmark scripts, TLA+ bounded model checking, and Tamarin symbolic proofs. The result is not a production standard, but it is a proof-carrying baseline for a new class of agent-native Web-service protocols.
